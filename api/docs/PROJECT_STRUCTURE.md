# Project Structure

This project follows a modular, feature-based architecture to ensure scalability, maintainability, and clear separation of concerns.

## Directory Overview

### `src/modules/`

Each feature (e.g., Auth, Users, Post, Article) is encapsulated within its own directory. This is the heart of the business logic.

- **`*.routes.ts`**: Defines the endpoints, attaches middleware, and applies Zod validation.
- **`*.controller.ts`**: Handles incoming HTTP requests, extracts validated data, calls the service, and returns standardized responses. Controllers should be thin.
- **`*.service.ts`**: Contains the core business logic, complex orchestrations, and high-level validation (e.g., checking if a user is banned).
- **`*.repository.ts`**: Direct interaction with the database using Prisma. This layer abstracts data access.
- **`*.validator.ts`**: Zod schemas for validating request bodies and URL parameters.
- **`index.ts`**: Public API for the module.

### `src/platform/`

Internal infrastructure and framework setup. These are core services shared across the entire application.

- **`Application/`**: Boots the system, registers modules, and configures the Hono server.
- **`database/`**: Prisma client initialization and health checks.
- **`cache/`**: Redis client setup and abstraction.
- **`config/`**: Configuration management using environment variables.
- **`http/`**: Hono server setup, middleware, and type definitions (`AppEnv`).
- **`logger/`**: Standardized logging using Pino.

### `src/shared/`

Shared utilities and constants that don't belong to a specific module or platform component.

- **`json/`**: Standardized success (`ApiResponse`) and error (`AppError`) classes.
- **`utils/`**: Helper functions (hashing, validation wrappers, etc.).
- **`constants/`**: Reusable constants like `HttpStatusCode`.
- **`types/`**: Global TypeScript interfaces and types.

### `prisma/`

Contains the database schema (`schema.prisma`) and historical migration files.

### `src/generated/`

Contains code generated by tools, primarily the Prisma client.

---

## Key Architectural Principles

1.  **Separation of Concerns**: Each layer (Route -> Controller -> Service -> Repository) has a single responsibility.
2.  **Modularization**: New features should be added as new modules in `src/modules/`.
3.  **Thin Controllers**: Controllers should only handle HTTP concerns and data extraction.
4.  **Fat Services**: Business logic belongs in services.
5.  **Repository Pattern**: Abstracting Prisma behind repositories allows for easier testing and potential future database swaps.
